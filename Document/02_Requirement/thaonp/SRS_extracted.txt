Use Case Specifications2.1 Authentication & OnboardingUC-101: New Account RegistrationPrimary ActorsGuest User (P03 - Lan Anh)Secondary ActorsNoneDescriptionAs a Guest User, I want a single-screen, friction-less registration process (email + password only) so that I can onboard and start using the platform immediately, achieving the BG-01 goal of registration in ≤ 30 seconds. (Related Story: LB-101)PreconditionsThe user must not have an existing account on LiveBox.PostconditionsA new User Account is successfully created in the PostgreSQL database.The system redirects the user to the primary Server view.The user is automatically logged in with a valid JWT Access Token.Normal Sequence/FlowThe user selects the "Register" option.The system displays the New Account Registration screen (single-screen flow).The user provides a valid Email and Password.The user clicks the "Sign Up" button.System validates the input (e.g., email format, password strength).The system hashes the password and persists the new account details in the database.System issues a new JWT Access Token (1h TTL) and a Refresh Token (7 days TTL), storing the Refresh Token in the database (C04).System logs in the user and redirects to the primary Server view.Alternative Sequences/FlowsAF 5.1: Invalid InputIf the email format is invalid or the password does not meet criteria, the system displays an inline error message (e.g., MSG02) next to the corresponding field. The user must correct the input before proceeding.AF 5.2: Existing EmailIf the provided email is already registered, the system displays a clear error message (e.g., "Email already in use") and prompts the user to log in or reset the password.UC-102: Login and Session PersistencePrimary ActorsServer Owner (P01), Active Member (P02), Guest User (P03)Secondary ActorsNoneDescriptionAs a returning LiveBox User, I want the system to securely handle session persistence using JWT and Refresh Tokens, so that I don't have to re-authenticate every time I open the application, supporting the BG-05 requirement for secure access. (Related Story: LB-102)PreconditionsUser account has been successfully created and verified.User has valid login credentials (email and password) or a valid Refresh Token.PostconditionsThe user is successfully authenticated and logged into the system.The system issues a valid JWT Access Token (1h) and Refresh Token (7 days).Normal Sequence/FlowA. Initial Login via CredentialsThe user accesses the LiveBox application.The system presents the Login Screen.The user enters Email and Password.The user clicks the "Log In" button.The system validates credentials against the PostgreSQL database.If valid, the system generates and returns a JWT Access Token (1h) and a Refresh Token (7 days), storing the Refresh Token in the database (C04).System redirects the user to the primary Server view.B. Session Persistence (Token Refresh Flow)The user opens the application and the local Access Token is expired (T > 1 hour).The client application attempts to exchange the stored Refresh Token for a new Access Token.System validates the Refresh Token against the database (C04).If valid, the system issues a new JWT Access Token and redirects the user to the Server view without re-prompting for credentials.Alternative Sequences/FlowsAF 5.1: Invalid Credentials (Authentication Failure)If the Email or Password is incorrect, the system displays the error message: "Invalid email or password. Please try again." (MSG05).AF 5.2: Token Revocation Failure (Security Check)If the Refresh Token is invalid or has been revoked (e.g., due to secure logout), the system rejects the token request and forces the user back to the Login Screen for explicit re-authentication.UC-103: Secure LogoutPrimary ActorsServer Owner (P01), Active Member (P02), Guest User (P03)Secondary ActorsNoneDescriptionAs a LiveBox User, I want a secure logout mechanism (LB-103) that revokes my Refresh Token (C04) and closes the WebSocket connection (G14) so that my session is fully terminated and unauthorized access is prevented (BG-05).PreconditionsUser is successfully logged in and has an active JWT Access Token and a valid Refresh Token persisted in the database (C04).PostconditionsThe user's active session is destroyed.The Refresh Token is revoked (deleted from DB).The system redirects the user to the Login Screen.Normal Sequence/FlowThe user clicks the "Logout" button.Client application sends a request to the backend with the Refresh Token.System validates the token, revokes the Refresh Token (C04) by deleting it from the PostgreSQL database (C03).System closes the active WebSocket connection (G14).System returns a successful logout response and clears the Access Token from the client.Client redirects the user to the Login Screen.Alternative Sequences/FlowsAF 3.1: Token Validation FailureIf the Refresh Token is already expired or invalid, the system still proceeds to clear local tokens and redirect to the Login Screen, ensuring a clean session termination and security integrity.2.2 Server ManagementUC-201: Server CreationPrimary ActorsServer Owner (P01)Secondary ActorsGuest User (P03)DescriptionAs a Server Owner (Anh Đạt - The system must implement JWT Refresh Token rotation with a 7-day validity period.Access Control: 100% of server-specific resources (channels, member lists) must be restricted to authenticated members of that specific server.Session Management: The system must provide a secure logout function that immediately revokes the Refresh Token in the database and terminates the WebSocket connection.Data Integrity: All data in transit must be encrypted using TLS 1.2 or higher (HTTPS/WSS).4.2.4 Reliability & AvailabilityUptime Target: The system shall maintain an availability of 99.5% during the Phase 1 demonstration period.Fault Tolerance: In the event of a WebSocket disconnection, the ReactJS client must implement an exponential backoff reconnection logic to restore real-time functionality without requiring a manual page refresh.Data Persistence: 100% of sent messages and server configurations must be stored in the PostgreSQL database with no data loss during standard operation.4.2.5 ScalabilityVertical Scaling: The Spring Boot backend and PostgreSQL database must be capable of vertical scaling (increasing CPU/RAM) on cloud providers (e.g., Render, Railway) to handle sudden spikes in user activity.Database Growth: The schema must support a growth trajectory of up to 10,000 users and 1,000,000 messages before requiring horizontal sharding or partitioning.4.2.6 MaintainabilityDeployment (Vibe Coding): The system must utilize a CI/CD pipeline (automated via GitHub Actions or cloud-native tools) to ensure new updates can be deployed to production in < 10 minutes.Monitoring & Logging: The backend must implement centralized logging (e.g., SLF4J/Logback) to track 100% of critical errors and authentication failures for rapid debugging.Code Documentation: The codebase must maintain a minimum of 80% documentation coverage for core API services to facilitate future Phase 2 development.4.2.7 CompatibilityBrowser Support: The system must be fully functional on the latest stable versions of Google Chrome, Mozilla Firefox, and Microsoft Edge.Responsive Design: The web application must remain usable on screen widths ranging from 1024px (Tablet/Small Laptop) to 1920px+ (Desktop).5.
