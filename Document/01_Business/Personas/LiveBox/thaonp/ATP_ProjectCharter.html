Project Description

1. Purpose (Business Case)

The LiveBox project aims to construct a highly scalable, real-time communication platform parallel to Discord. The objective is to replace disparate legacy polling constraints with a persistent Client-Server architecture. By guaranteeing ultra-low latency (< 200ms) and ensuring uninterrupted UX through Single Page Application (SPA) technologies, the platform attempts to serve 10k+ concurrent active subscribers seamlessly without state desynchronization.

2. Scope Definitions

In-Scope (Deliverables):
- Provisioning infrastructure for Server & Channel structural entities (Text & Basic Voice Channels).
- Sub-<200ms Real-time Messaging engine governed by WebSocket protocols & Spring Boot Message Broker.
- Robust Auth workflows relying on JWT encapsulated within HttpOnly cookies via Spring Security.
- Deep UI optimizations utilizing React component re-rendering and Optimistic UI update patterns.

Out-of-Scope (Exclusions for Phase 1):
- Complex automated Bot integrations or third-party webhooks.
- Advanced Video conferencing routing protocols (WebRTC multiparty arrays).
- Custom animated Emoji processing pipelines.

3. Key Output Deliverables

1. Modern responsive Frontend Client built efficiently via ReactJS (TypeScript).
2. Robust Backend Web API deployed on Java Spring Boot environment.
3. Centralized authoritative Database System utilizing PostgreSQL ensuring complete ACID compliance.
4. Unified Technical Contracts (Agile User Stories, OpenAPI specs, ERDs, and Stakeholder Matrix mapping).

6. Risk Management Plan

Identified Architecture Risks Impact BA/Dev Mitigation Strategy
WebSocket connection drops destroying UI State High Implementation of robust Auto-reconnect strategies on the React Client. Leveraging Optimistic UI to queue pending message payloads while network stabilizes.
Security flaws during Authentication flows High Strict compliance with Spring Security interceptors. Mandating BCrypt hashing for persistency and intercepting all 401 exceptions on Frontend SPA routers automatically.
Data inconsistency during heavy concurrent writes Medium Deferring to PostgreSQL's native locking mechanisms ensuring transactions remain ACID compliant across active server workspaces globally.
