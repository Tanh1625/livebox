Stakeholder Matrix

Survey & Analysis of LiveBox Project Stakeholders
Version: v1.0 (Baseline)

Power/Interest Grid Analysis

Mapping the influence and engagement levels of the key individuals, target users, and technical teams that impact the success of the LiveBox real-time ecosystem.

Stakeholder / Role Power Interest Objectives & Expectations BA Management Strategy
Sponsor / Decider
CTO / Project Sponsor 		High High Requires a scalable architecture that guarantees < 200ms latency and supports 10k+ concurrent users. Demands zero data loss via ACID-compliant PostgreSQL transactions. Manage Closely: Present technical blueprints (Architecture, DB schemas) for approval. Provide KPI dashboards on system load tests and security audits.
Key User
Community Admin (Alex) 		High High Needs a robust workspace provisioning flow without F5 page reloads. Demands explicit Role-Based Access Control (RBAC) tools to manage Server/Channel members. Keep Satisfied: Involve them in early React Component testing. Ensure UI state synchronizes perfectly to Backend mutations to guarantee a seamless SPA experience.
End User
Authenticated Member (Jamie) 		Low High Relies entirely on instant message delivery. Despises network lag or frozen UI frames. Demands obvious visual indicators (Optimistic UI) when a message is pending server ACK. Keep Informed & Monitor: Beta-test the WebSocket messaging flow intensively. Prioritize UX edge-case handling for network degradation or dropped socket connections.
IT / Implementer
Development Team (FE/BE) 		Med High Requires clear un-ambiguous technical specifications: DTO mappings, WebSocket sub-protocols, JWT payload definitions, and PostgreSQL entity relations. Consult & Collaborate: Provide granular Agile User Stories. Run frequent grooming sessions to align the React (TypeScript) and Spring Boot teams on identical API contracts.
Quality Assurance
QA / Tester Team 		Med High Responsible for ensuring system reliability and quality across real-time messaging and voice communication. Requires clear acceptance criteria, test scenarios for WebSocket flows, concurrency handling, and failure recovery (e.g., dropped connections, retry mechanisms). Focuses on detecting race conditions and validating end-to-end user experience. Consult & Validate: Define detailed test cases from User Stories. Collaborate closely with Dev team to validate API contracts and WebSocket events. Perform regression, load, and integration testing to ensure system stability under high concurrency conditions.
External User
Anonymous Visitor (Sam) 		Low Med Expects industry-standard security regarding credential handling (BCrypt hashing). Easily frustrated by unhandled 401 exceptions or clunky login redirects. Monitor (Design Focus): Ask QA to simulate guest navigation. Stress-test the Spring Security filter chain to ensure smooth redirects rather than generic server error pages.

📝 BA Note: Due to the heavy reliance on real-time Client-Server streaming, the Development Team operates as a central pivot. Any discrepancy between the React frontend logic and the Spring Boot backend WebSocket routing will immediately cascade into catastrophic UX failure for the Key/End Users. Strict API consensus is mandatory.
