User Personas - LiveBox Project

Derived from Agile User Stories. This serves as the North Star for UI/UX design, System Architecture, and Backlog prioritization in the LiveBox development lifecycle.

👑

Alex - "The Organizer"

Key User | Community Admin / Server Owner

"I need to provision server instances and organize channels without jumping through convoluted administrative hoops."

Behavior & Personality

System administrator mindset. Manages multiple workspaces simultaneously. Demands efficient workflows, granular access control, and immediate feedback when applying state mutations to the channel registry.

Pain Points

  • Provisioning structural entities (servers/channels) often involves tedious form submissions and disjointed UX.
  • Inconsistent UI state synchronization requiring hard refreshes (F5) to see new channels.
  • Lack of clear mapping between domain entities (Text vs Voice channels).

System Objectives

  • Provisioning Submit DTOs to provision Server Workspaces and persist DB records seamlessly.
  • Reactivity UI auto-re-renders component trees upon state mutations in the Sidebar channel registry.

Jamie - "The Socializer"

End User | Authenticated User / Member

"I expect instant message delivery. If the app lags or drops connection silently, it ruins the conversational flow."

Behavior & Personality

High-frequency interaction. Expects extreme low-latency (< 200ms) data exchange. Highly sensitive to UX degradation during network bottlenecks or unacknowledged WebSocket payloads.

Pain Points

  • HTTP polling overhead causing network latency and battery drain.
  • Ambiguous message states when API requests are stuck in-flight.
  • Losing connection context without automated client-side reconnect strategies.

System Objectives

  • Real-time Message broadcasting to active subscribers via WebSocket protocol.
  • Optimistic UI Render pending states immediately upon Enter key event, awaiting server ACK for persistence confirmation.

Sam - "The Explorer"

External User | Anonymous Visitor / Guest

"If the registration flow is clunky or I hit an ambiguous 401 error page, I'll probably just close the tab."

Behavior & Personality

First-time evaluator. Focuses on onboarding friction. Values clear constraints (e.g., regex validation messages) and expects industry-standard security practices for handling credentials.

Pain Points

  • Submitting registration payloads only to be rejected by obscure backend validation rules.
  • Clicking protected routes and getting abruptly dropped into generic error screens.
  • Unclear token lifecycle management leading to unexpected session terminations.

System Objectives

  • Security Robust hashing (BCrypt) and secure JWT issuance encapsulated in HttpOnly cookies.
  • Resilience Graceful interception of 401 Unauthorized exceptions, triggering automated redirects to the Login route.
